# Quick 软件需求规范说明文档

## 一、项目背景

- 小企业内部Web后端项目

## 二、技术选型

### 2.1 配置管理

- `github.com/spf13/viper` + 环境变量

  > 支持 YAML 配置文件与环境变量混合加载，便于多环境部署。

### 2.2 认证与权限控制

- **JWT 认证**：`github.com/golang-jwt/jwt/v5`

  > 用于生成和验证 JSON Web Token，实现无状态身份认证。

- **RBAC 权限控制**：

    - `github.com/casbin/casbin/v2`

    - `github.com/casbin/gin-casbin/v2`

      > 基于角色的访问控制（RBAC），通过策略文件或数据库动态管理权限，中间件自动拦截并校验请求。

### 2.3 Web 框架与中间件

- **Web 框架**：`github.com/gin-gonic/gin`

- **参数校验**：`github.com/go-playground/validator/v10`

  > 结构体字段标签校验（如 `validate:"required,email"`），集成于 Gin 绑定流程。

- **统一响应格式**：自定义 `Response` 结构体 + 辅助函数

  > 标准化 API 输出格式（code/data/msg），提升前端对接体验。

- **全局错误处理**：

    - 自定义中间件（捕获 panic、统一错误码）

    - `github.com/getsentry/sentry-go`

      > 实时上报未处理异常至 Sentry，支持上下文附加（用户 ID、请求路径等）。

### 2.4 日志系统

- **结构化日志**：

    - `go.uber.org/zap`

    - `github.com/gin-contrib/zap`（Gin 集成）

      > 高性能、结构化日志记录，支持 JSON 格式输出，便于日志收集（如 ELK、Loki）。

### 2.5 数据存储

- **PostgreSQL 驱动**：`github.com/jackc/pgx/v5`

  > 高性能原生 PostgreSQL 驱动，支持连接池、批量操作、监听/通知等高级特性。

- **SQL 代码生成**：`sqlc`（[https://sqlc.dev](https://sqlc.dev/)）

  > 通过编写原生 SQL 自动生成类型安全的 Go 数据访问层（DAO），避免手写重复的 `Scan`/`Exec` 逻辑，与 `pgx/v5` 深度集成。

- **Redis 客户端**：`github.com/redis/go-redis/v9`

  > 用于缓存、分布式锁、会话存储等场景，支持集群、哨兵和管道操作。

## 四、功能需求

### 4.1 需求管理

Quick 需支持以下功能：

- 需求输入
- 需求类别自定义
- 明确区分市场需求 → 系统需求 → 软件/硬件需求 → 单元需求
- 类型标签管理：标记需求为“功能安全需求”“信息安全需求”“法规需求”“性能需求”等。
- **端到端追溯链**：
    - 市场需求 → 系统架构 → 软件需求 → 软件代码 → 测试用例
    - 市场需求 → 系统架构 → 硬件需求 → 电路设计 → 硬件测试
    - **影响分析**：当某条硬件需求变更（如更换芯片），自动识别受影响的软件模块和测试项。
    - **覆盖率报告**：确保每条安全相关需求都有对应的测试用例覆盖（满足 ISO 26262 要求）。

用户可新建项目，设置名称、内部代号、客户、介绍、新建/指定已有团队（包括项目经理、系统工程师）

### 4.5 团队管理

#### **1.** 项目经理（Project Manager）

- 负责整体项目计划、资源协调、进度控制；
- 确保项目符合 ASPICE 的“项目管理”过程（MAN.3）；
- 协调功能安全、网络安全、质量等跨领域目标。

#### **2.** 系统架构师（System Architect）

- 定义 BCM 的系统架构（硬件+软件）；
- 支持 ISO 26262 中的“技术安全概念”（TSC）和“系统架构设计”（SY.2）；
- 支持 ISO/SAE 21434 的“网络安全架构设计”（RUA.5 / SEC.2）；
- 需熟悉 AUTOSAR 架构（）。

### **3.** 功能安全工程师（Functional Safety Engineer）

- 主导 HARA 分析、ASIL 等级分配；
- 编写安全目标、安全需求、验证安全机制；
- 参与 FMEA、FTA、FMEDA 等分析；
- 对接 ISO 26262 各阶段（概念→开发→生产）；
- 若项目不强制 ISO 26262，也建议设置此角色进行风险评估。

### **4.** 网络安全工程师（Cybersecurity Engineer）

- 主导 TARA（Threat Analysis and Risk Assessment）；
- 制定网络安全目标与需求（CSMS）；
- 设计安全通信、密钥管理、安全启动等机制；
- 满足 ISO/SAE 21434 的 RUA、SEC、VAL 等过程域；
- 与功能安全团队协同处理“安全 vs 安全”冲突（Safety vs Security）。

### **5.** 软件开发工程师（Software Developers）

- 分模块开发（如驱动层、应用层、通信层）；
- 遵循 ASPICE 的 SWE.1–SWE.6（软件开发过程）；
- 使用 MISRA C / AUTOSAR / Safe coding practices；
- 支持单元测试、集成测试（需工具链支持，如 VectorCAST、Tessy）。

### **6.** 硬件开发工程师（Hardware Engineer）

- 负责 PCB、电源、EMC、接口电路设计；
- 支持 ISO 26262 的硬件安全需求（如单点故障度量、PMHF）；
- 进行硬件可靠性分析（如 FMEDA）；
- 配合 EMC 实验室完成抗扰度测试。

### **7.** 测试与验证工程师（Test & Validation Engineer）

- 制定测试策略（HIL/SIL/MIL）；
- 执行功能安全验证（FSR 验证）、网络安全渗透测试；
- 覆盖 ASPICE 的 SYS.4、SYS.5、SWE.4–SWE.6；
- 使用 CANoe、dSPACE、Vector 工具链。

### **8.** 配置管理与变更控制工程师（CM Engineer）

- 建立版本控制系统（Git + ALM 如 Polarion/Jama）；
- 管理需求-设计-代码-测试的双向追溯（traceability）；
- 满足 ASPICE 的 SUP.8（配置管理）和 SUP.9（变更请求管理）。

### **9.** 质量保证工程师（QA Engineer）

- 监督开发过程是否符合 ASPICE L2/L3 要求；
- 组织过程审计、差距分析、改进措施；
- 推动 V-model 开发流程落地。

### **10.** 需求工程师（Requirements Engineer）

- 收集客户/OEM 需求，转化为系统/软硬件需求；
- 使用 DOORS 或 Jama 管理需求层级；
- 确保需求可测试、可追溯、无歧义；
- 支持 ASPICE 的 SYS.2 / SYS.3。
